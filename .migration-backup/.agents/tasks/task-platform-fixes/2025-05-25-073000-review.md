# Platform hardening: security, infrastructure, and business logic

A sweeping audit-response change addressing 67 findings across security, data integrity, observability, and business logic for an Indian SMB SaaS (CRM + WhatsApp + e-commerce + booking). The change adds Zod-based field allowlisting to close mass assignment, CSRF double-submit cookies, Meta webhook HMAC verification, structured logging with request IDs, cursor-based pagination, a Postgres-backed job queue for broadcast sends, booking conflict detection, DPDP-compliant data export, and a vitest test harness with 18 initial tests. Infrastructure gains include body size limits, graceful shutdown, deep health checks, auth-table indexes, and LRU eviction on the rate limiter.

**Watch for:** Zod schemas use `z.any()` on every field (confirmed) — the allowlist stops unknown keys but admits garbage values. CSRF cookie omits the `Secure` flag (confirmed). The data export endpoint loads all user data into memory unbounded (confirmed). The job queue `processJobs` claims a row without a row-level lock, creating a race under concurrent workers (likely).

## High-level view

The mass-assignment fix works structurally — `patchContactSchema.parse(raw)` strips keys not in the schema — but the per-field validators are all `z.any()`, so the type-level protection against malformed values (wrong types, over-length strings, SQL-hostile payloads) is deferred. The fields that reach the DB are constrained to a known set; the values within those fields are not.

CSRF protection uses a standard double-submit cookie pattern but has two gaps: the token cookie is set without `Secure` (leaks over HTTP), and the check is skipped entirely on a client's first request because the cookie doesn't exist yet (the code checks `needsCheck && token`, where `token` is undefined for new sessions).

Meta webhook signature verification rejects forged payloads when `META_APP_SECRET` is configured, but degrades to a `logger.warn` and open-door processing when the env var is absent. In production with the env var unset, any party that knows the webhook URL can inject messages.

The job queue's `processJobs` function does `UPDATE ... SET status='processing' WHERE status='pending' ... RETURNING` which is safe under a single worker but races if multiple Node processes poll concurrently — Postgres will update all matching rows in one shot without a `LIMIT 1` or row lock, and multiple workers will claim the same batch.

The data-export endpoint (`GET /export/my-data`) runs 10 sequential unbounded `SELECT *` queries for a single user. A user with 100k messages will produce a multi-hundred-MB JSON response, risking OOM and timeouts. No streaming, no pagination, no size guard.

Cursor-based pagination is opt-in: if the client omits `cursor`/`limit` params, endpoints still return full unbounded result sets. The pagination fix only helps clients that explicitly adopt it; the underlying DoS vector remains for legacy callers.

8. **Broadcast job serial send with no progress** — consider batching or parallel sends (respecting Meta rate limits) and writing intermediate progress to the broadcast row.

10. **`requireVerifiedEmail` issues a DB query on every request** — cache the verification status in the session or use a short TTL in-memory lookup to avoid per-request overhead.
11. **`escapeSqlLike` not applied to all ILIKE queries** — verify that every user-facing search path uses the utility (admin search uses it, but audit other routes).

## Zod allowlisting: structure without depth

The validators in `server/lib/validators.ts` are purely structural allowlists:

```ts
export const patchContactSchema = z.object({
  name: z.any(),
  phone: z.any(),
  email: z.any(),
  // ...
}).partial();
```

`.parse(raw)` strips unexpected keys (closing mass assignment) but `z.any()` means `{ name: 42 }`, `{ phone: [1,2,3] }`, or `{ email: { "$ne": "" } }` all pass. Drizzle will coerce or reject at the Postgres layer, surfacing an unhandled 500 from a constraint violation rather than a clean 400 with a meaningful message. The structural pattern is correct; the next step is tightening each field to proper types with length/range bounds.

## CSRF: first-request bypass and missing Secure

The double-submit implementation in `server/middleware/csrf.ts` checks `if (needsCheck && token)` — but `token` comes from `getCookie(c, "csrf_token")` which returns `undefined` for a brand-new session. The cookie is only set *after* `await next()`, so the very first POST/PATCH/DELETE from a new client passes through with no check. An attacker who can force a victim into a fresh session (clear cookies via XSS or incognito lure) gets one unprotected state-changing request.

The cookie is also set without `secure: true`:

```ts
setCookie(c, "csrf_token", token, {
  path: "/",
  sameSite: "Lax",
  httpOnly: false,
});
```

Without `Secure`, the cookie transmits over plain HTTP, so a network observer on any HTTP redirect can capture it and construct a valid CSRF attack.

## Meta webhook: soft-fail on missing secret

```ts
if (metaAppSecret) {
  // verify signature
} else {
  logger.warn('META_APP_SECRET not set -- skipping signature verification');
}
```

Fail-open pattern. If the env var is accidentally deleted during a deploy, the webhook processes completely unauthenticated payloads. In production, this should return 503 or 401 — a missing secret means the system is misconfigured and shouldn't process potentially forged messages.

## Job queue: single-worker assumption baked in

```ts
const [job] = await db.update(jobQueue)
  .set({ status: 'processing', ... })
  .where(and(eq(jobQueue.status, 'pending'), lte(jobQueue.scheduledFor, new Date())))
  .returning();
```

This translates to `UPDATE ... WHERE status='pending' AND scheduled_for <= $1 RETURNING *` without `LIMIT 1`. Postgres will update *all* pending jobs in one shot. The code destructures `[job]` to grab the first, but under concurrent workers both claim the same batch. The idiomatic fix:

```sql
UPDATE job_queue SET status='processing' ...
WHERE id = (SELECT id FROM job_queue WHERE status='pending'
            ORDER BY scheduled_for LIMIT 1 FOR UPDATE SKIP LOCKED)
RETURNING *;
```

Additionally, `stopJobWorker()` sets `polling = false` but doesn't await in-flight `processJobs()`. A job mid-execution during shutdown may be interrupted, leaving it stuck in `processing` status with no retry path on next startup.

## Data export: memory and timeout risk

`GET /export/my-data` runs 10 sequential queries with no `LIMIT`:

```ts
const messages = await db.select().from(message).where(eq(message.ownerId, userId));
```

A power user with 50k messages loads ~50MB into a single Node array before serialization. Mitigation: stream NDJSON, generate the export asynchronously as a file, or cap each table's rows with pagination.

## Booking conflict: TOCTOU gap

In `POST /bookings` the conflict check and the insert are separate statements outside a transaction:

```ts
const [conflict] = await db.select(...).from(booking)
  .where(and(/* same date+time */)).limit(1);
if (conflict) return 409;
// ... later ...
const [row] = await db.insert(booking).values({ ... }).returning();
```

Two concurrent requests for the same slot both pass the SELECT (neither has inserted yet) and both succeed at the INSERT. Fix: a partial unique index on `(owner_id, booking_date, booking_time) WHERE status NOT IN ('cancelled','no_show')`, or wrap both in a serializable transaction.

## Pagination: backward-compatible but unprotected

`wantsPagination(c)` checks whether the client sends `cursor` or `limit`. If neither is present, the route falls through to the unbounded query. The DoS vector from the audit (dump entire contact table) is still live for any caller that omits those params. Adding a hard server cap (`.limit(1000)`) on the non-paginated path would close this without breaking existing clients.

## Rate limiter LRU eviction

Eviction at 100k buckets sorts the entire Map synchronously:

```ts
const entries = Array.from(buckets.entries());
entries.sort((a, b) => a[1].resetAt - b[1].resetAt);
```

O(n log n) on the hot path will spike latency for the unlucky request that triggers it. A min-heap or LRU linked-list would keep eviction O(1). Unlikely to hit at MVP scale but worth noting for growth.

## Test coverage: depth limited

The 18 tests establish vitest infrastructure. Crypto tests are strong (round-trip, IV randomness, corruption, Unicode). Webhook and billing tests are thinner — they recreate route logic inline rather than importing the actual handlers, so they verify behavior of a copy, not of production code. A bug fixed in `webhooks.ts` wouldn't necessarily fail `webhooks.test.ts`.

Not tested: CSRF middleware, pagination edge cases, job queue, booking conflict detection, data export, coupon atomic redemption.
