# Platform hardening: security, infrastructure, and business logic (v2)

The eight primary concerns from the v1 review have been addressed. Zod schemas now enforce proper types with length/range bounds. CSRF rejects first requests without a cookie (forcing a GET first). Meta webhook fails closed in production. The job queue uses `FOR UPDATE SKIP LOCKED` for atomic claims. Data export caps at 10,000 rows per table. Non-paginated paths have server-side limits. Booking conflict detection is wrapped in a transaction.

**Watch for:** Zod `.parse()` throws unhandled — surfaces as a generic 500 to clients instead of a 400 with field errors (confirmed). `GET /tasks` non-paginated path still has no `.limit()` cap (confirmed).

**Verdict**: APPROVED

## High-level view

The Zod validators are now properly typed — `z.string().max(255)`, `z.number().int()`, `z.enum([...])`. However, `.parse()` is called without a try/catch in any route handler, and the global `onError` doesn't distinguish `ZodError` from unexpected failures. A malformed PATCH body gets a 500 with `{ error: "Internal server error" }` rather than a 400 with actionable field-level details. This is a UX/DX gap, not a security gap.

The one remaining unbounded query is `GET /tasks`, which uses `db.query.task.findMany(...)` without a `limit` property on the non-paginated branch. Every other list endpoint now has `.limit(1000)`. Minor omission — tasks are unlikely to hit thousands of rows per user at MVP scale — but it breaks the otherwise consistent pattern.

## Zod validation error propagation

Every PATCH route calls `.parse()` directly:

```ts
const body = patchContactSchema.parse(raw);
```

When validation fails, Zod throws a `ZodError`. The global error handler catches it:

```ts
app.onError((err, c) => {
  logger.error({ method: c.req.method, path: c.req.path, err }, 'Unhandled error');
  if (err instanceof Error && /not.?found/i.test(err.message)) {
    return c.json({ error: err.message }, 404);
  }
  return c.json({ error: "Internal server error" }, 500);
});
```

No branch recognizes `ZodError`, so the client sees `500 { error: "Internal server error" }` for a request like `PATCH /contacts/:id { name: 42 }`. The fix is either to catch `ZodError` in `onError` and return a 400 with the issues array, or to use `.safeParse()` in each handler and return 400 explicitly. The former is one line; the latter is more explicit per-route.

## Tasks endpoint: lone unbounded query

```ts
app.get("/tasks", async (c) => {
  if (!wantsPagination(c)) {
    const rows = await db.query.task.findMany({
      where: eq(task.ownerId, c.var.userId),
      orderBy: [sql`${task.dueAt} ASC NULLS LAST`, desc(task.createdAt)],
      with: { contact: true },
    } as any);  // <-- no limit
    return c.json(rows);
  }
```

Compare with the identical pattern in the deals endpoint, which passes `limit: 1000`. The omission is likely a copy-paste oversight.
